    Replace the current RESULT_OK handler inside the `deleteLauncher` callback with a re-query pattern that determines which IDs were actually deleted before propagating to onDeleteConfirmed.

    The `repository` and `coroutineScope` are already available at the composable body level (repository via `remember`, coroutineScope via `rememberCoroutineScope()`). Add `rememberCoroutineScope()` at the top of the ReviewRoute composable body (before the LaunchedEffect blocks), then update the deleteLauncher callback:

    ```kotlin
    // Add at composable body level (before deleteLauncher):
    val coroutineScope = rememberCoroutineScope()

    // Replace the existing deleteLauncher callback with:
    val deleteLauncher = rememberLauncherForActivityResult(
        contract = ActivityResultContracts.StartIntentSenderForResult(),
    ) { result ->
        if (DeleteRequestCoordinator.isDeleteConfirmed(result.resultCode)) {
            val submittedIds = viewModel.uiState.value.selectedPhotoIds
            coroutineScope.launch {
                // Re-query to find which IDs are actually gone from MediaStore.
                // loadReviewPhotos returns only IDs still present; absent ones were deleted.
                val stillPresentIds = repository
                    .loadReviewPhotos(submittedIds.toList())
                    .map { it.id }
                    .toSet()
                val actuallyDeleted = submittedIds - stillPresentIds

                if (actuallyDeleted.isNotEmpty()) {
                    // Normal path: at least some deletions succeeded.
                    viewModel.onDeleteConfirmed(actuallyDeleted)
                }
                // If actuallyDeleted is empty: all submitted IDs still present in MediaStore
                // (complete failure). Do not navigate away — review state is preserved intact.
                // The user sees no change and can retry or cancel.
            }
        }
        // Cancel (RESULT_CANCELED): do nothing — review state unchanged.
    }
    ```

    Add import: `kotlinx.coroutines.launch`

    The re-query runs on the scope's default dispatcher (Main) calling a suspend function. The `loadReviewPhotos` implementation must switch to IO internally (it already does via the existing MediaStorePhotoRepository implementation — verify the existing call at line 78 compiles and runs without issues).

    Do NOT change the LaunchedEffect(stagedPhotoIds) block or the RequestDelete event handler. Only the RESULT_OK branch of deleteLauncher changes.
